  • Application, Java JDK: Optimizing best-of-2 work stealing queue selection on OpenJDK
    • /* Description: The idea is to keep last successfully stolen queue as one of best of two candidates on the OpenJDK to reduce the number of steals and total failed steal attempts. */
    • This bug exists in OpenJDK 1.8 and later, comfirmed by Zhengyu Gu and Thomas Schatzl from Redhat (Merged List).
  • Application, Cloudsuite: Running web-search client container problem
    • /* Description: Client container under web-search benchmark fails to start with error message. Curl is missing in the official Dockerfile. */
    • The corresponding fix to the bug has been merged into offical branch (Merged List).
  • Hypervisor, Xen: Long tail latency caused by rate-limit in Xen credit2
    • /* Description: When one I/O-bound VM shares the physical CPU with one CPU-bound VM and the hypervisor is using credit2 scheduler, the average and tail latency of the I/O-bound VM will increase significantly. The reason is caused by the default ratelimit in the Xen scheduler. Similar issue is also available in Xen credit1 scheduler. */
    • This bug exists in Xen 4.8, comfirmed by Dario Faggioli(Xen Develop Mailing List).
  • Application, Java JDK: Compiler version test failed of SPECJVM2008 benchmarks on OpenJDK
    • /* Description: Compiler version detection is failed when executing SPECJVM2008 benchmarks on customized OpenJDK. */
    • This issue exists when executing SPECJVM2008 on OpenJDK 1.7, 1.8.
  • Application, Java JDK: Load imbalance of Tasks among GCTaskThreads during the Young GC
    • /* Description: GCTasks from GCTaskQueue are not distributed evenly among the GCTaskThreads, which causes the load imbalance among GCTaskThreads and thus prolongs the Young GC time. */
    • This bug exists in OpenJDK 1.8.
  • Hypervisor, Xen: Bugs existing Xen's credit scheduler cause long tail latency issues
    • /* Description: If a VCPU is in UNDER state and wakes up from sleep, it will be boosted in csched_vcpu_wake(). However, the boosting is successful only when __runq_tickle() preempts the current VCPU. It is possible that csched_acct() can run between csched_vcpu_wake() and __runq_tickle(), which will sometimes change the BOOST state back to UNDER if credit is greater than 0. That contributes to the long tail latency. */
    • This bug exists in Xen 4.3, 4.4, 4.5, comfirmed by George Dunlap(Xen Develop Mailing List).
  • Hypervisor, Xen: Bugs existing Xen's credit scheduler cause long tail latency issues
    • /* Description: In csched_acct() (by default every 30ms), a VCPU stops earning credits and is removed from the active CPU list(in __csched_vcpu_acct_stop_locked) if its credit is larger than the upper bound. However, csched_tick() will only put the current VCPU back to the active list instead of all inactive VCPUs. As such, the I/O-bound VM will be unable to be boosted and have very long latency. It takes at least one time slice (e.g., 30ms) before the I/O VM is activated and starts to receive credits. */
    • This bug exists in Xen 4.3, 4.4, 4.5, comfirmed by George Dunlap(Xen Develop Mailing List).
  • OS, Linux: Linux process vruntime accounting mistakenly in virtualized environment
    • /* Description: Linux CFS relies on delta_exec to charge the vruntime of processes. The variable delta_exec is the difference of a process starts and stops running on a CPU. This works well in physical machine. However, in virtual machine under capped resources, some processes might be accounted with inaccurate vruntime. */
    • This bug exists in Linux 3.18.x to 4.7.x, confirmed by Boris Ostrovsky(Xen Develop Mailing List), added patch by Juergen Gross(Xen Develop Mailing List, Linux Kernel Mailing List, Patchwork List), fixed in Linux 4.8.x by adding xen_steal_clock in drivers/xen/time.c.
  • Application, libmemcached: libmemcached configure bug with memaslap enabled
    • /* Description: memaslap cannot be found when libmemcached is compiled with option --enable-memaslap. The configure file does not hint when corresponding library that memaslap need is missing. */
    • This bug exists in libmemcached-1.0.18.
  • Hypervisor, Xen: mistakenly wake in Xen's credit scheduler
    • /* Description: VCPU that runs CPU-intensive workload could be woken mistakenly due to the unawareness of CPU affinity which could make significant I/O performance loss. */
    • This bug exists in Xen 4.3, 4.4, 4.5, 4.6, confirmed by Dario Faggioli from Citrix(Xen Develop Mailing List).
